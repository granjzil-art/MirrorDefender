# Mirror 塔防 · 协作规范与边界提案

> 在你给出的 4 条硬性边界（可拓展 / 参数化 / 模块化 / 文档维护）基础上，补充我认为有利于健壮性、编辑易用性、人机协作效率的规范。
> 版本：v0.1 · 2026-07-15

---

## 一、你已定的四条（我的落地方式）

### 1. 可拓展
- 一切"会变的东西"走**接口/策略**：地块形状(`ITileShape`)、镜子效果(`IMirrorEffect`)、索敌(`ITargetingStrategy`)、攻击(`IAttackStrategy`)。
- 六边形坐标系用立方体坐标封装，换成三角/方格密铺时只替换 Grid 层。
- 数据与逻辑分离：塔/单位/关卡用 Godot Resource(.tres) 定义，加内容不改代码。

### 2. 参数化
- 所有可调效果参数用 `@export`（GDScript）暴露到检视面板，支持**运行时**调节。
- 参数分组用 `@export_group`，加范围 `@export_range` 防止手滑填非法值。
- 全局数值集中到 `GameConfig` 资源，便于统一打磨与平衡。

### 3. 模块化
- 目录即模块：`/systems/grid`、`/systems/mirror`、`/systems/camera`、`/units`、`/buildings`、`/ui`、`/input`。
- 模块间通过**信号(signal)/事件总线**通信，禁止跨模块直接持有引用（降耦合）。
- 每个模块可独立开关（feature flag），便于隔离调试。

### 4. 持续维护文档（每改代码即改文档）
- 每个模块一个 `README.md`：功能概述 / 关键架构 / 函数索引(函数名→一句话职责) / 参数说明 / 已知限制。
- 文档放在模块目录内，就近维护。
- 提交约定：改了代码的 commit 必须带上对应文档改动，否则视为未完成。

## 二、我补充提议的规范

### 5. 设计评估四步流程（把你的要求固化为工作流）
> 每个功能都走这四步，不跳步：
1. **理解对齐**：复述我理解的设计点，列出逻辑缺口/踩坑点/不明确处 → 向你提问，**不擅自改方案**。
2. **方案选型**：有多解时列优劣表 → 请你拍板。
3. **执行计划 + 验收标准**：拆任务、定"怎样算做完"。
4. **验收交付**：按标准自检，告诉你【用法入口 / 文件结构 / 参数位置】。

### 6. 命名与代码规范
- 命名：类 `PascalCase`、变量/函数 `snake_case`、信号 `snake_case` 过去式（如 `unit_died`）、常量 `UPPER_CASE`。
- 一个脚本一个职责，超过 ~300 行考虑拆分。
- 魔法数字一律进 `GameConfig` 或 `@export`，代码里不硬编码。

### 7. 调试可视化（Gizmo）
- 核心系统提供可开关的调试绘制：六边形坐标、寻路路径、索敌范围、镜面对称轴、光线反射路径。
- 用一个全局 `DebugDraw` 开关统一控制，方便你边玩边看内部状态。

### 8. 数值与平衡外置
- 所有平衡数值（塔数值、单位数值、经济曲线、波次）走**数据表(.tres/CSV/JSON)**，策划可直接改表，不碰代码。

### 9. 场景/资产约定
- KayKit 模型统一封装成 Prefab 场景(.tscn)，逻辑挂在根节点，美术可替换而不影响逻辑。
- 命名前缀区分：`Tile_` / `Bld_` / `Unit_` / `FX_` / `UI_`。

### 10. 变更留痕
- 每次功能落地后，在 `Docs/CHANGELOG.md` 追加一行：日期 / 模块 / 改了什么 / 影响面。
- 大的方案决策记入 `Docs/DECISIONS.md`（谁、为什么、否决了哪些备选），避免反复推翻。

### 11. 提交前自检清单（Definition of Done）
- [ ] 功能按验收标准可跑
- [ ] 新增参数已 `@export` 且分组
- [ ] 对应模块 README 已更新
- [ ] CHANGELOG 已追加
- [ ] 无跨模块硬引用、无遗留魔法数字
- [ ] 调试可视化可开关

### 12. 性能红线（针对镜子机制）
- 折射反射次数上限（默认 ≤2），镜像递归代数上限（默认 ≤1），面镜层数上限。
- 大量单位用对象池 + 流场寻路，避免每单位跑独立 A*。

## 三、协作节奏建议
- **一次一个模块**：不并行开多个大系统，每个模块走完四步流程再进下一个。
- **先灰盒后美术**：用色块/占位把玩法跑通验证好玩，再套 KayKit 美术。
- **你负责设计决策，我负责实现与提案**：涉及玩法手感/数值取向/主题表达的选择，我列选项你拍板；纯工程实现我直接做。

> 以上第 5–12 条为新增提议，若认可我就写进项目根的 `CONTRIBUTING.md` 作为长期约束。
